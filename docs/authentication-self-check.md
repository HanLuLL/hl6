# 认证与 Android 闭环自查清单

本清单覆盖 HL6 当前邮箱密码认证架构中可预期的注册、登录、已有账户激活、密码重置、认证邮件、反向代理、运行时 URL 和 Android 原生会话场景。它用于发布前验收、线上故障排查和事故复盘。

它不能替代安全评审或实际监控。遇到未列出的症状时，应保留请求时间、HTTP 状态、浏览器或客户端版本和服务端错误日志，再沿认证状态流排查。

## 使用规则

1. 使用隔离测试账号和测试邮箱。不得把真实密码、`AUTH_PASSWORD_PEPPER`、SMTP 密码、JWT、`X-HL6-Client-Key` 或邮件原始令牌写入工单、日志或截图。
2. 每项记录执行人、时间、部署版本、访问域名、客户端类型、结果和证据链接。
3. 浏览器与 Android 要分别验收。浏览器使用 HttpOnly Cookie；Android 使用 Bearer 会话和通讯密钥，二者不可互相代替。
4. 任一“阻断项”失败时停止发布；修复根因后从相关场景的第一项重新执行。
5. 对未知邮箱只验证响应保持中性，绝不通过响应内容、时延或页面文案推断账户是否存在。

## 认证状态模型

| 状态 | 用户可执行操作 | 正确结果 |
| --- | --- | --- |
| 新邮箱，无凭据 | 注册 | 收到注册验证邮件，设置密码后创建用户和会话 |
| 已注册，密码已设置 | 登录、忘记密码 | 登录成功或收到重置邮件 |
| 旧用户，需要激活 | 激活；登录页应给出明确引导 | 收到激活邮件，设置密码后保留原 `users.id` 和业务数据 |
| 已封禁用户 | 登录或设置密码 | 进入封禁/申诉流程，不能访问普通受保护资源 |
| 已过期封禁用户 | 登录或设置密码 | 自动恢复正常用户并建立可用会话 |
| 无效、过期或已消费令牌 | 设置密码 | 显示链接无效或过期，绝不设置密码或建立会话 |

## 阻断项：部署与基础配置

- [ ] `AUTH_PASSWORD_PEPPER_ID` 和非空、高熵 `AUTH_PASSWORD_PEPPER` 已配置，且所有应用实例一致。
- [ ] 会话签名密钥可用且跨实例一致；滚动部署时实例之间可互相验证会话。
- [ ] `FRONTEND_URL` 已配置为面向用户的 HTTPS 域名；多域名部署时填写全部正式域名。
- [ ] `BACKEND_URL`、`VITE_API_BASE_URL`、`ALLOWED_ORIGINS` 与实际浏览器 Origin 一致，不含尾随路径或未预期域名。
- [ ] 优先采用同站点部署，例如 `app.example.com` 与 `api.example.com`。跨注册域 Cookie 需单独完成 `SameSite=None; Secure`、CSRF 和第三方 Cookie 策略评审。
- [ ] HTTPS 入口正确传递 `X-Forwarded-Proto: https`；`hl6_session` Cookie 具有 `HttpOnly`、正确的 `Secure` 和预期的 `SameSite` 属性。
- [ ] SMTP 已启用，发件地址、TLS、认证信息和发信测试均通过。首个管理员注册前必须完成这一步。
- [ ] 新安装、旧库升级、已完成 cutover 的实例分别确认 `auth.local.enabled` 正确。旧库未完成 `hl6-admin auth preflight` 与 `hl6-admin auth cutover` 前，本地认证不可用是保护行为。
- [ ] 数据库、应用和邮件系统的时间误差在认证令牌 TTL 内可接受；当前令牌有效期为 30 分钟。

## 阻断项：反向代理与真实客户端 IP

- [ ] Gin 仅信任明确配置的反向代理 IPv4、IPv6 或 CIDR，并使用其转发的客户端 IP。
- [ ] 未配置可信代理时，应用只使用连接端 Remote Address，不采信用户可伪造的 `X-Forwarded-For`。
- [ ] Nginx、Traefik、CDN 或负载均衡器将可信客户端 IP 链路完整传递到应用；多层代理时确认地址解析顺序与实际拓扑一致。
- [ ] 从两个不同网络连续发起认证请求：同一邮箱和同一 IP 限流仍生效；不同用户不再因共享代理地址共同触发 `429`。
- [ ] `429` 在 Web 和 Android 都显示“请求过于频繁”，不被显示为密码错误、网络错误或未登录。

## 阻断项：运行时 URL、CORS 与 CSRF

- [ ] 静态环境 URL、数据库 URL 列表和管理员确认状态的优先级符合部署设计；公共认证请求不能把任意 Host 或转发头持久化为可信 URL。
- [ ] 多域名确认后，从每个已配置前端域名请求注册、激活和密码重置都能生成该域名对应的邮件链接。
- [ ] URL 确认签名只覆盖规范化的受信任 URL 配置，不随当前请求选择的 active host 改变。
- [ ] CORS 和 Cookie-CSRF 信任来源使用同一份经过规范化验证的前端 URL 列表。动态更新数据库 URL 后，受信任 Origin 的写请求无需重启即可工作。
- [ ] 对每个正式域名验证：登录、登出、更新资料、认领子域名和 DNS 写操作均不被 CORS 或 CSRF 错误拦截。
- [ ] 非受信任 Origin 不能携带 Cookie 对任何写接口执行操作。

## Web：注册流程

### 正常注册

- [ ] 有效、未注册邮箱提交 `POST /api/v1/auth/registration/request` 后收到 `202` 中性响应并进入查收邮件页。
- [ ] 邮件主题、正文、按钮和备用链接与 `locale` 一致；未传 locale 时遵循 `Accept-Language`，再回退英文。
- [ ] 邮件链接指向受信任的 HTTPS 前端域名和 `/set-password?token=...`，没有开放重定向、查询参数污染或错误 Host。
- [ ] 打开邮件链接后，页面把令牌读入内存并立即从可见地址栏和历史记录移除；响应使用 `Referrer-Policy: no-referrer` 与 `Cache-Control: no-store`。
- [ ] 使用 8 至 128 个 Unicode 码点的密码和相同确认密码完成设置；用户、默认用户组、积分余额、推荐关系和会话必须一致创建。
- [ ] 设置完成后刷新 Dashboard，`/auth/me` 返回实际用户与实际积分余额，不得长期显示临时 `credits: 0` 缓存。

### 输入、策略与重试边界

- [ ] 空邮箱、非法邮箱、过长邮箱和无效 IDN 邮箱返回 `400 error.invalidRequestBody`，页面保留输入并展示可理解错误。
- [ ] 注册关闭、允许名单、阻止名单、精确 IDNA 匹配、重复条目和通配符拒绝行为与后台设置一致。
- [ ] 非法推荐码返回 `400`；合法、无推荐、失效和邀请自己都不破坏账户创建事务。
- [ ] 重复点击或网络重试不会创建多个用户或多份积分；多个短期邮件令牌的消费和失效规则明确且可观测。
- [ ] 已激活账户提交注册时保持中性，不泄露账户存在；页面只能说明“如账户符合条件，将收到邮件”。
- [ ] 已存在且待激活账户提交注册时，在保持中性响应的前提下发送激活链接或提供明确的“激活已有账户”恢复路径，不能假称已发送邮件。
- [ ] SMTP、URL 配置或发信失败时，管理员能在邮件日志定位原因；用户页提供安全重试和联系支持提示。

## Web：登录流程

### 正常登录

- [ ] 已激活用户使用正确邮箱和密码登录。浏览器获得 Cookie 并进入 Dashboard；Android 获得 `access_token` 与 `expires_in`，不依赖浏览器 Cookie。
- [ ] 登录后立即刷新 `/auth/me` 或查询缓存，用户组、管理员身份、封禁字段和积分与服务端一致。
- [ ] 已轮换 password pepper 的旧哈希登录后按设计重新哈希，失败时不破坏既有凭据或会话。
- [ ] 主动退出后服务端递增 session version；此前浏览器 Cookie 和 Android Bearer token 都失效。

### 错误凭据与异常状态

- [ ] 错误邮箱、错误密码、空密码和未激活账户由登录页展示安全错误或引导，绝不整页刷新、丢失输入或显示“缺少令牌”。
- [ ] 公共认证端点的 `401` 只解析错误响应，不清除 Android 已保存的会话；只有受保护资源 `401` 才执行会话失效处理。
- [ ] 待激活旧用户在登录页有明确激活入口；若产品选择登录时自动发送激活邮件，必须保持防枚举、遵守限流并记录安全事件。
- [ ] 已封禁用户进入 `/banned`，仍可查看封禁信息、提交申诉和退出，但不能访问其他受保护资源。
- [ ] 临时封禁已到期时，下次登录自动恢复账户。
- [ ] 服务不可用、数据库错误、密码服务不可用、CORS 失败和网络超时分别显示正确且可行动的提示。

## Web：已有账户激活与密码重置

- [ ] `ActivationRequiredAt != nil` 的旧账户请求 `POST /auth/activation/request` 后收到中性 `202` 与激活邮件。
- [ ] 未知邮箱、已激活账号、邮件服务不可用和用户记录缺失不泄露账户状态；管理员可通过安全事件或邮件日志区分原因。
- [ ] 激活邮件使用 `/set-password`，密码重置使用 `/reset-password`；查收邮件页必须识别 `registration`、`activation`、`reset` 三种 mode，并显示对应标题、说明、重试和返回路径。
- [ ] 激活后复用原始 `users.id`，保留域名、DNS、积分、头像、封禁、通知、支付、申诉和审计数据。
- [ ] 链接重复、过期、并发使用或后发与先发链接竞争时，只有一个密码变更成功；其他链接返回中性无效令牌错误，不能覆盖新密码。
- [ ] 激活完成或密码重置后，旧会话和未消费的恢复令牌按设计失效。

## 共享令牌与密码设置

- [ ] 原始令牌只存在邮件链接和当前页面内存中；数据库、应用日志、审计日志、错误报告和浏览器 Referrer 均不保存它。
- [ ] 数据库只保存 SHA-256 令牌哈希，令牌具有目的、用户或邮箱绑定、过期时间和消费时间。
- [ ] 令牌在执行昂贵的密码哈希前只能由一个请求原子消费；密码服务短暂不可用时存在安全的重新签发路径。
- [ ] 密码不合规则、两次密码不一致、缺少令牌和令牌过期分别得到正确错误；密码不写入 URL、日志或本地存储。

## Android：构建、App Links 与原生会话

### 构建前置

- [ ] Android 以 `web/` React 应用为唯一认证 UI，不维护重复的原生登录、注册或激活页面。
- [ ] 通讯域名、通讯密钥、应用 ID、版本和显示名称有效；通讯密钥只在 APK 构建时注入，不能出现在公开日志或页面。
- [ ] `@capacitor/app`、Secure Storage、Capacitor WebView 和 Android Manifest 的版本互相兼容。
- [ ] 每次认证页面或路由变更后重新构建 Web 并执行 Capacitor sync，确保 APK 资产包含最新页面。

### Android App Links

- [ ] 使用 HTTPS Android App Links，不使用可被其他应用抢占的自定义 scheme。
- [ ] 邮件前端域名与 APK 构建时允许的域名一致；Manifest 的 `VIEW` intent filter 只匹配该 HTTPS host 和 `/set-password`、`/reset-password` 所需路径。
- [ ] 声明 `android:autoVerify="true"`；发布域名的 `/.well-known/assetlinks.json` 无重定向、可公开访问，并包含正确包名和签名证书 SHA-256 指纹。
- [ ] 在真实已签名 APK 上测试：应用已安装时邮件链接打开 APK，未安装时安全回退到网页。
- [ ] 支持冷启动、后台恢复和前台 URL 打开事件；只接受预期 HTTPS host、路径和单一 token 查询参数，拒绝未知 host、路径、片段和多值参数。
- [ ] 路由到共享 `/set-password` 或 `/reset-password` 后，令牌立即从可见 URL 移除，并仅保留在当前页面的内存状态。

### 原生认证行为

- [ ] Android 登录、设置密码和重置或激活完成请求都带 `X-HL6-Client-Key`；受保护请求同时带匹配的 `Authorization: Bearer`。
- [ ] 服务端只在有效通讯密钥存在时返回 `access_token`；Web 浏览器从不把该 token 暴露给 JavaScript。
- [ ] 登录或设置密码成功后，`access_token` 写入 Android Keystore 支持的 Secure Storage，并同步内存状态；进程重启后可恢复。
- [ ] Android 点击邮件链接、在 App 内设置密码后直接进入 Dashboard 或封禁页，不要求再次手工登录。
- [ ] 错误密码、无效链接和公共端点 `401` 不清除已有原生会话；受保护资源 `401`、主动退出和通讯密钥撤销才清除会话。
- [ ] 通讯密钥轮换或撤销时，旧 APK 只能获得受限强制更新信息，不能继续调用普通 API；新 APK 更新后能正常注册、登录和激活。

## 邮件、日志与可观测性

- [ ] 注册、激活、重置邮件均记录类型、收件人脱敏信息、发送时间、状态和不含令牌的失败摘要。
- [ ] 邮件发送失败、URL 未确认、SMTP 禁用、域名策略拒绝、限流、无效令牌和错误凭据均有可查询的安全事件或日志。
- [ ] 日志中不出现明文密码、password hash、原始认证令牌、会话 JWT、SMTP 密码或 Android 通讯密钥。
- [ ] 管理员能区分“没有收到邮件”“未完成 cutover”“URL 配置未确认”“SMTP 故障”“被限流”等问题，而无需读取用户敏感凭据。

## 症状到根因速查

| 现象 | 首先检查 | 常见根因 | 处置 |
| --- | --- | --- | --- |
| 输错密码后页面刷新或输入消失 | `/auth/login` 的 401 处理 | 公共登录 401 被当成会话失效 | 仅对受保护请求执行跳转或清会话 |
| Android 错误密码显示缺少令牌 | 原生 API 401 处理 | 公共登录 401 清除了 native token | 保留错误体，不清现有会话 |
| 多名用户同时 429 | 服务端看到相同 `RemoteAddr` | 代理地址被当作客户端 IP | 配置可信代理并安全解析真实 IP |
| 显示已发信但收不到 | 邮件日志、SMTP、URL 确认 | SMTP、URL、旧账号状态或中性响应设计 | 保持防枚举并提供管理员诊断 |
| 一个域名能发邮件，另一个不能 | `frontend_urls` 和确认签名 | active host 被纳入签名 | 仅签名规范化配置列表 |
| 登录成功但写操作 403 | Origin 和 CSRF 信任列表 | DB 运行时 URL 未同步到 middleware | CORS 和 CSRF 共享动态 URL 列表 |
| Android 邮件链接进入浏览器 | Manifest、`assetlinks.json`、URL listener | 无 App Link 或未处理冷/热启动 | 配置 App Links 并路由到共享页面 |
| App 内设置密码后仍未登录 | 请求头、`access_token`、Secure Storage | 浏览器完成或未持久化 native token | 在 APK 内完成并写入安全存储 |
| 所有本地认证接口 503 | `auth.local.enabled` 和 cutover | 旧库尚未完成迁移 | 完成预检和 cutover，或显示迁移状态 |
