# Feature Specification: 域名多 DNS 供应商支持

**Feature Branch**: `001-add-dns-providers`  
**Created**: 2026-04-11  
**Status**: Draft  
**Input**: User description: "为域名添加多 DNS 供应商支持。原只支持 cloudflare，目标支持以下平台作为供应商：DNSPod、阿里云 DNS、DNS.com、DNSLA、DnsDun、西部数码、华为云 DNS、百度智能云 DNS、Amazon Route 53、Google Cloud DNS。" 

## Clarifications

### Session 2026-04-11

- Q: 当管理员修改一个“已有 DNS 记录”的域名的供应商绑定时，系统应采用哪种策略？ → A: 采用自动迁移，并改为后台异步任务；允许部分失败且支持手动重试。
- Q: 迁移任务执行期间，域名 DNS 写操作如何处理？ → A: 迁移期间域名只读，允许查询，禁止新增/修改/删除记录。
- Q: 域名生效供应商何时切换到目标供应商？ → A: 创建迁移任务后立即切换到目标供应商。
- Q: 发起域名跨供应商迁移时，系统应该迁移哪些 DNS 记录？ → A: 仅迁移平台数据库已托管记录（dns_records）。
- Q: 管理员点击“手动重试”时，重试范围应如何定义？ → A: 仅重试失败项（按记录粒度增量重试）。
- Q: 迁移成功后是否自动清理源供应商记录？ → A: 默认不自动清理；提供管理员可选清理功能。
- Q: 迁移任务结束但存在失败项时，域名写入策略如何处理？ → A: 任务结束即解除只读并使用新供应商；未迁移项由管理员手动重试。
- Q: 同一域名在迁移进行中时，是否允许再次发起迁移任务？ → A: 允许重复提交，但后提交任务进入队列，前一个完成后自动执行（串行）。
- Q: 失败项手动重试时，若目标供应商存在同名同类型但内容不同记录，系统如何处理？ → A: 直接覆盖目标记录为平台期望值（upsert）。
- Q: 同一域名有排队迁移任务时，若前序任务部分失败，后续任务如何处理？ → A: 不论前序任务结果，后续排队任务都自动开始执行。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 管理员接入多供应商 (Priority: P1)

作为平台管理员，我可以在系统中新增并验证不同 DNS 供应商账号，使平台能为不同域名选择对应供应商进行记录管理。

**Why this priority**: 没有供应商接入能力，后续任何用户侧 DNS 管理都无法落地。

**Independent Test**: 在仅实现“供应商接入与验证”功能的情况下，管理员应可成功新增任一受支持供应商账号并通过连通性校验，系统可保存其可用状态。

**Acceptance Scenarios**:

1. **Given** 管理员已登录，**When** 提交某受支持供应商的有效凭据，**Then** 系统保存该供应商连接并标记为“可用”。
2. **Given** 管理员已登录，**When** 提交无效或权限不足的凭据，**Then** 系统拒绝保存并返回可理解的失败原因。
3. **Given** 某供应商暂不可用，**When** 管理员执行连接测试，**Then** 系统显示失败状态且不影响其他供应商连接。

---

### User Story 2 - 用户按域名选择供应商管理记录 (Priority: P1)

作为普通用户，我可以在自己有权限的域名下使用该域名绑定的供应商进行 DNS 记录增删改查，不需要关心底层供应商差异。

**Why this priority**: 用户实际价值来自“跨供应商统一管理子域名与记录”，这是核心业务能力。

**Independent Test**: 在已接入至少 1 家非 Cloudflare 供应商时，用户可在目标域名下完成记录列表、新增、修改、删除全流程。

**Acceptance Scenarios**:

1. **Given** 域名已绑定供应商且用户有权限，**When** 用户创建符合规则的记录，**Then** 记录成功生效并可在列表中看到。
2. **Given** 用户尝试创建与现有记录冲突的内容，**When** 提交操作，**Then** 系统阻止提交并给出明确冲突原因。
3. **Given** 用户在同一账号下管理不同域名，**When** 切换到不同域名，**Then** 系统自动使用对应域名绑定的供应商处理请求。

---

### User Story 3 - 运维可观测与故障隔离 (Priority: P2)

作为平台运维/管理员，我可以追踪每次供应商调用的结果与失败类型，以便快速定位单一供应商故障并隔离影响。

**Why this priority**: 多供应商场景下，稳定性和可定位性决定了平台可运营性。

**Independent Test**: 在模拟单一供应商失败时，系统可记录失败事件、显示失败原因并保证其他供应商域名操作仍可继续。

**Acceptance Scenarios**:

1. **Given** 某供应商接口报错，**When** 用户发起记录变更，**Then** 系统记录失败事件并返回可读错误信息。
2. **Given** 另一供应商处于正常状态，**When** 用户在该供应商域名执行同类操作，**Then** 操作不受前述失败影响并可成功完成。

### Edge Cases

- 当目标供应商账号凭据被撤销或过期时，系统如何阻止继续写入并提示管理员重新授权。
- 当供应商返回限流或瞬时网络错误时，系统如何避免重复写入导致的脏数据。
- 当某供应商不支持特定记录能力或命名约束时，系统如何在提交前提示不可用而不是提交后失败。
- 当供应商侧已存在同名记录但与平台状态不一致时，系统如何提示冲突并支持人工处理。
- 当单个用户可管理多个供应商域名时，系统如何确保跨域名请求不会串用错误凭据。
- 当域名发生跨供应商迁移且记录量较大时，系统如何通过后台任务反馈进度、保留失败项并支持手动重试。
- 当域名处于迁移中状态时，系统如何一致地拒绝写操作并向用户返回可理解提示。
- 当创建迁移任务后立即切换生效供应商时，系统如何避免用户误以为记录已全部迁移完成。
- 当源供应商存在平台未托管记录时，系统如何避免在迁移中误处理这些记录。
- 当管理员多次触发手动重试时，系统如何确保仅针对失败项执行并避免重复覆盖已成功迁移记录。
- 当管理员执行源供应商清理时，系统如何确保仅清理平台托管记录并可追溯操作主体与结果。
- 当迁移任务结束但仍有失败项时，系统如何持续提示风险并引导管理员补齐重试。
- 当同一域名存在排队中的迁移任务时，系统如何展示队列顺序、当前任务状态与预计执行顺序。
- 当失败项重试触发覆盖写入时，系统如何记录被覆盖对象与覆盖结果，避免不可审计的静默变更。
- 当前序迁移任务部分失败但后续任务继续自动执行时，系统如何控制级联失败风险并保持任务可追溯性。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 支持为域名配置 DNS 供应商，并覆盖 Cloudflare、DNSPod、阿里云 DNS、DNS.com、DNSLA、西部数码、华为云 DNS、百度智能云 DNS、Amazon Route 53、Google Cloud DNS。
- **FR-002**: 系统 MUST 允许管理员为每个供应商保存、更新、停用和连通性验证凭据。
- **FR-003**: 系统 MUST 为每个域名绑定且仅绑定一个生效中的 DNS 供应商。
- **FR-004**: 用户 MUST 能在授权域名下以统一交互完成记录的查询、新增、修改、删除，且沿用平台既有权限模型。
- **FR-005**: 系统 MUST 在执行记录变更前完成供应商能力校验与参数校验，并在不满足约束时阻止提交。
- **FR-006**: 系统 MUST 对所有供应商返回统一错误语义（如鉴权失败、权限不足、记录冲突、限流、供应商不可达），用于前端展示和重试决策。
- **FR-007**: 系统 MUST 记录供应商操作审计日志，至少包含操作主体、目标域名、目标记录、请求动作、结果状态与失败原因。
- **FR-008**: 系统 MUST 在单一供应商异常时保持故障隔离，不影响其他供应商域名的 DNS 操作。
- **FR-009**: 系统 MUST 提供供应商状态可视化，管理员可查看每个供应商连接状态与最近失败事件。
- **FR-010**: 首个发布版本 MUST 一次性上线本期范围内全部新增供应商，且每家均达到统一的完整读写能力，不允许按供应商分批或降级交付。
- **FR-011**: DnsDun MUST 不纳入本期交付范围，且本期系统 MUST 不提供 DnsDun 作为可选供应商；后续如需支持，作为独立需求进入新迭代。
- **FR-012**: 本期跨供应商记录类型 MUST 仅保证 A、AAAA、CNAME、TXT 四类记录的一致可用能力。
- **FR-013**: 当管理员变更域名供应商且该域名存在 DNS 记录时，系统 MUST 创建后台迁移任务，将既有 A、AAAA、CNAME、TXT 记录迁移到目标供应商，而非同步阻塞完成。
- **FR-014**: 迁移任务 MUST 允许部分失败，不因单条失败回滚全部已成功项，并且 MUST 提供失败项可见性与管理员手动重试能力。
- **FR-015**: 迁移任务运行期间，域名 MUST 进入只读状态；系统 MUST 允许记录查询并拒绝新增、修改、删除操作，直至迁移任务结束。
- **FR-016**: 当迁移任务创建成功时，系统 MUST 立即将域名生效供应商切换为目标供应商，并通过显式迁移状态提示“迁移进行中/部分失败/待重试”。
- **FR-017**: 域名迁移任务 MUST 仅处理平台数据库已托管的 A、AAAA、CNAME、TXT 记录；源供应商中未被平台托管的记录 MUST 保持不变且不纳入自动迁移。
- **FR-018**: 迁移任务的手动重试 MUST 仅针对失败项执行，已成功迁移项 MUST 不被重复执行或覆盖。
- **FR-019**: 迁移流程 MUST 默认不自动删除源供应商记录；系统 MUST 提供管理员可选的“源供应商清理”操作，且该清理范围仅限平台托管记录并记录审计日志。
- **FR-020**: 迁移任务结束（无论成功或部分失败）后，域名 MUST 解除只读并继续以目标供应商作为解析供应商；失败项 MUST 保持可见并支持管理员手动重试直至处理完成。
- **FR-021**: 同一域名在已有迁移任务运行时，系统 MUST 允许新迁移请求进入队列并串行执行；任一时刻同域名 MUST 仅有一个运行中的迁移任务，后续任务在前序任务终态后自动开始。
- **FR-022**: 失败项手动重试遇到目标供应商同名同类型冲突时，系统 MUST 以平台期望值覆盖目标记录（upsert），并记录覆盖前后关键信息用于审计追踪。
- **FR-023**: 同一域名的迁移队列 MUST 在前序任务达到任意终态（成功或部分失败）后自动启动后续任务，不因前序任务失败项而自动暂停或取消队列。

### Key Entities *(include if feature involves data)*

- **DNS 供应商**: 表示可接入的平台类型与其能力描述（名称、可用状态、能力标签、故障状态）。
- **供应商凭据配置**: 表示管理员维护的连接信息与有效状态（所属供应商、脱敏标识、启停状态、最后验证结果）。
- **域名供应商绑定**: 表示一个域名当前生效的 DNS 供应商关联关系与变更历史。
- **域名 DNS 迁移任务**: 表示域名跨供应商切换时的异步迁移过程与结果（总数、成功数、失败项、重试状态）。
- **供应商操作事件**: 表示每次 DNS 调用行为的审计与故障信息（动作、对象、结果、错误分类、时间）。
- **记录能力策略**: 表示不同供应商支持范围与平台统一规则之间的映射关系。

## Assumptions

- 平台现有用户权限、积分控制、域名/子域名归属机制保持不变，仅在 DNS 执行层扩展供应商。
- 各供应商均以“可配置凭据并按域名执行记录管理”为目标能力，但不同供应商在鉴权、配额、记录约束上存在差异。
- 产品默认优先保障跨供应商一致体验，而非暴露所有供应商独有高级能力。
- 对供应商官方政策限制（如资质、配额、区域限制）采取“显式提示 + 管理员可见状态”处理。
- DnsDun 需求在本期明确排除，后续是否支持由独立需求决定。

## Dependencies

- 本期目标供应商提供可用于生产环境的官方接口能力与账号权限。
- 平台管理员具备对应供应商账号、权限与必要资质（若平台有资质限制）。
- 现有域名与 DNS 记录数据可与新增供应商绑定模型兼容迁移。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 管理员可在 10 分钟内完成任一受支持供应商的接入并得到明确可用/不可用判定。
- **SC-002**: 在已接入供应商的域名上，90% 以上常规记录变更操作可在 1 分钟内完成并向用户返回明确结果。
- **SC-003**: 多供应商并存场景下，单一供应商故障不会导致其他供应商域名操作失败，跨供应商故障串扰率低于 1%。
- **SC-004**: 上线后 30 天内，因“供应商差异导致用户无法判断失败原因”的工单占 DNS 工单总量比例低于 10%。
- **SC-005**: 管理员可在 3 分钟内从系统内定位任一失败操作的主体、对象和失败分类，无需查看外部系统。
